# Rust Release Automation and Versioning Strategies 2025

**Research Date:** December 2025
**Target:** Production-grade release automation for Rust CLIs

---

## Executive Summary

This research synthesizes best practices from leading Rust CLI projects (ripgrep, bat, fd, starship) and modern release automation tools. The recommended stack combines **release-plz** for versioning, **cargo-dist** for binary distribution, and **git-cliff** for changelog generation, enabling fully automated releases with minimal manual intervention.

## 1. Versioning Tools Comparison

### 1.1 Tool Evaluation Matrix

| Tool | Approach | Best For | Maturity | GitHub Actions | CI-Native |
|------|----------|----------|----------|-----------------|-----------|
| **release-plz** | Registry-based version detection | Pure Rust projects | Stable | Yes (Action available) | Yes |
| **cargo-release** | Git tag-based versioning | CLI workflows | Stable | Limited | Manual |
| **semantic-release** | Conventional commits (npm ecosystem) | Mixed stacks | Stable | Yes | Yes |
| **release-please** | Google's approach via GitHub Actions | Any language | Stable | Yes | Yes |

### 1.2 Release-plz (Recommended for Rust)

**Repository:** https://github.com/release-plz/release-plz
**Documentation:** https://release-plz.dev/
**crates.io:** https://crates.io/crates/release-plz

#### Key Advantages
- **Rust-native:** Designed specifically for Cargo projects
- **Smart versioning:** Compares local packages with crates.io registry (not git tags)
- **Zero configuration:** Works out-of-the-box with Cargo.toml
- **Workspace support:** Native support for Cargo workspaces
- **Integration:** https://github.com/release-plz/action - Official GitHub Action

#### How It Works
1. Analyzes commits since last release using Conventional Commits
2. Detects API breaking changes via cargo-semver-checks
3. Creates release PR with:
   - Updated `Cargo.toml` versions
   - Generated `CHANGELOG.md` (via git-cliff)
   - Updated `Cargo.lock`
4. Merge PR → release-plz automatically:
   - Creates git tags
   - Publishes to crates.io
   - Creates GitHub releases

#### Configuration Example
```toml
# release-plz.toml
[workspace]
changelog_file = "CHANGELOG.md"
changelog_template = "..."

[workspace.changelog]
trim = true

[package]
semver_check = true  # Verify API compatibility
```

#### GitHub Action Integration
```yaml
- name: Install Rust
  uses: dtolnay/rust-toolchain@stable

- name: Run release-plz
  uses: release-plz/action@v0.5
  with:
    token: ${{ secrets.GITHUB_TOKEN }}
```

**Industry Endorsement:** Niko Matsakis noted that "release-plz and trusted publishing and cargo-semver-checks [are] getting to a very cool place."

### 1.3 Automatic Version Bumping Strategies

#### Conventional Commits Mapping
```
fix(parser):      → PATCH version bump (0.0.X)
feat(cli):        → MINOR version bump (0.X.0)
feat!: breaking   → MAJOR version bump (X.0.0)
BREAKING CHANGE   → MAJOR version bump (X.0.0)
```

#### Rust Tools for Version Calculation
- **CocoGitto:** https://lib.rs/crates/cocogitto - Full Conventional Commits + SemVer toolbox
- **convco:** https://crates.io/crates/convco - Calculates next version from commits
- **conventional_commits_next_version:** Monorepo-aware version detection

### 1.4 Changelog Generation with git-cliff

**Repository:** https://github.com/orhun/git-cliff
**Documentation:** https://git-cliff.org/
**crates.io:** https://lib.rs/crates/git-cliff

#### Features
- Highly customizable template system
- Conventional Commits support out-of-the-box
- Regex-powered custom parsers for any commit format
- Available as CLI, npm package, and Rust library

#### Configuration Example
```toml
# cliff.toml
[changelog]
# changelog template
template = "..."
# changelog footer
footer = "<!-- generated by git-cliff -->"
trim = true

[git]
# Parse the commits based on https://www.conventionalcommits.org
conventional_commits = true
filter_unconventional = true

[git.commit_parsers]
[[git.commit_parsers]]
message = "^feat"
group = "Features"
[[git.commit_parsers]]
message = "^fix"
group = "Bug Fixes"
```

#### Usage in CI
```bash
git-cliff --output CHANGELOG.md --tag "$(cat Cargo.toml | grep '^version' | head -1 | cut -d' ' -f3)"
```

**Integration:** Used by release-plz automatically for changelog generation

## 2. Binary Distribution Strategy

### 2.1 cargo-dist (Recommended for Binary Distribution)

**Repository:** https://github.com/axodotdev/cargo-dist
**Documentation:** https://axodotdev.github.io/cargo-dist/
**Blog Post:** https://axo-blog.pages.dev/2023/02/cargo-dist
**Latest Release:** v0.30.2+ (October 2025)

#### What It Does
cargo-dist is "cargo build but for building final distributable artifacts." It:
1. Compiles binaries for all target platforms (13+ targets)
2. Creates platform-specific archives
3. Generates SHA256 checksums
4. Creates GitHub releases with artifacts
5. Generates install scripts
6. Supports GitHub Artifact Attestations

#### Supported Archive Formats
- **`.zip`** - Default for Windows (now compressed in v0.30.0+)
- **`.tar.gz`** - TAR with gzip compression
- **`.tar.xz`** - TAR with xz compression (Unix default)
- **`.tar.zstd`** - TAR with Zstandard compression

#### Platform Support (Example: 13 Target Matrix)
```
Linux:
  - x86_64-unknown-linux-gnu
  - x86_64-unknown-linux-musl
  - aarch64-unknown-linux-gnu
  - arm-unknown-linux-gnueabihf
  - i686-unknown-linux-gnu

macOS:
  - x86_64-apple-darwin (Intel)
  - aarch64-apple-darwin (Apple Silicon)

Windows:
  - x86_64-pc-windows-msvc
  - aarch64-pc-windows-msvc
  - i686-pc-windows-msvc
  - x86_64-pc-windows-gnu

Other:
  - s390x (IBM mainframe)
  - powerpc64le
```

#### GitHub Actions Integration
```yaml
- name: Install Rust
  uses: dtolnay/rust-toolchain@stable
  with:
    targets: x86_64-unknown-linux-gnu,aarch64-apple-darwin

- name: Generate CI
  run: cargo install cargo-dist && cargo dist generate-ci

# This generates a complete release workflow in .github/workflows/
```

#### Automatic CI Generation
```bash
cargo dist generate-ci
```
Creates `.github/workflows/release.yml` with:
- Platform matrix builds
- Archive creation
- Checksum generation
- GitHub release creation
- Installer script generation

#### Shell Installer (macOS/Linux)


#### PowerShell Installer (Windows)
```powershell

```

#### Package Managers
- **Cargo:** `cargo install project-name`
- **Homebrew:** `brew install project-name` (via tap)
- **apt/apt-get:** Via custom PPA or in distributions
- **Pacman:** Via AUR or official repos
- **Winget:** Windows Package Manager

### 2.3 Homebrew Distribution Automation

**Reference:** https://kawamurakazushi.com/20200217-publishing-a-rust-cli-to-homebrew/

#### Setup: Create Homebrew Tap
```bash
# Create repository: homebrew-<tap-name>
# e.g., homebrew-tools for `brew tap user/tools`
git init homebrew-tools
```

#### Formula Template
```ruby
class MyProject < Formula
  desc "Project description"
  homepage "https://github.com/user/project"
  url "https://github.com/user/project/releases/download/v#{version}/my-project-v#{version}-x86_64-apple-darwin.tar.gz"
  sha256 "..."
  license "MIT"

  depends_on "openssl" => :build

  def install
    bin.install "my-project"
    man1.install "my-project.1"
  end
end
```

#### Automation with tap-release
**Repository:** https://github.com/toolmantim/tap-release

```yaml
# .github/tap-release.yml
formula: Formula/my-project.rb
tap: user/homebrew-tools
```

When you publish a GitHub release, tap-release automatically updates the formula.

## 3. Automated Release Workflows

### 3.1 Ripgrep Release Workflow

**Workflow File:** https://github.com/BurntSushi/ripgrep/blob/master/.github/workflows/release.yml
**Release Checklist:** https://github.com/BurntSushi/ripgrep/blob/master/RELEASE-CHECKLIST.md

#### Two-Stage Process

**Stage 1: Prepare Release**
- Update version in `Cargo.toml`
- Run `cargo update` for dependencies
- Review `Cargo.lock` changes
- Check for semver-incompatible updates with `cargo outdated`
- Create and tag release

**Stage 2: Build & Distribute**
Triggers on version tag matching `[0-9]+.[0-9]+.[0-9]+`

```yaml
name: Release
on:
  push:
    tags:
      - '[0-9]+.[0-9]+.[0-9]+'

jobs:
  create-release:
    # Verify git tag matches Cargo.toml version
    # Create draft release

  build:
    strategy:
      matrix:
        targets:
          - x86_64-unknown-linux-gnu
          - x86_64-apple-darwin
          - aarch64-apple-darwin
          - x86_64-pc-windows-msvc
          - aarch64-pc-windows-msvc
          # ... 9 more targets
    steps:
      - name: Build
        run: cargo build --release
      - name: Strip binaries
        run: strip target/release/rg
      - name: Generate completions
        run: cargo run --release -- --generate {bash,fish,zsh,powershell} > completions
      - name: Create man page
        run: cargo run --release -- --help | help2man -N -o rg.1 -
      - name: Archive
        run: tar czf rg-$TAG-$TARGET.tar.gz -C target/release rg completions rg.1
```

#### Build Optimizations
- Full LTO (Link Time Optimization) enabled
- Static PCRE2 support compiled in
- Binaries stripped for size reduction

#### Artifact Generation
- Binary (stripped)
- Shell completions (bash, fish, zsh, PowerShell)
- Man pages
- Debian packages (.deb)
- SHA256 checksums

### 3.2 Starship Release Workflow

#### Key Features
- **Release-please integration:** `googleapis/release-please-action@v4`
- **Multi-platform matrix:** 11 target platforms
- **Code signing:**
  - Windows: SignPath for binary signing
  - macOS: Apple notarization
- **Package manager updates:**
  - Crates.io publication
  - Homebrew formula updates
  - Winget manifest updates
  - Chocolatey package updates
- **Documentation:**
  - Crowdin translation PR merging
  - Documentation site deployment

#### Workflow Stages

```yaml
  build:

            target: x86_64-apple-darwin
          - os: macos-latest
            target: aarch64-apple-darwin
          - os: windows-latest
            target: x86_64-pc-windows-msvc
          # ... 7 more targets

  sign-windows:
    # Uses SignPath for code signing

  notarize-macos:
    # Apple notarization workflow

  publish-packages:
    # Update Homebrew, Winget, Chocolatey
```

### 3.3 Bat and Fd Release Workflows

**Bat Workflow:** https://github.com/sharkdp/bat/blob/master/.github/workflows/CICD.yml
**Fd Workflow:** https://github.com/sharkdp/fd/blob/master/.github/workflows/CICD.yml

#### Common Pattern (sharkdp projects)

Both bat and fd use unified CICD.yml workflow with these stages:

1. **Extract Metadata**
   ```yaml
   crate_metadata:
     outputs:
       name: ${{ steps.metadata.outputs.name }}
       version: ${{ steps.metadata.outputs.version }}
       maintainer: ${{ steps.metadata.outputs.maintainer }}
       msrv: ${{ steps.metadata.outputs.msrv }}
   ```

2. **Code Quality Gates**
   - `cargo fmt --check` (ensure_cargo_fmt)
   - `cargo clippy` with `-D warnings` (lint_check)
   - MSRV testing
   - Security audit with cargo-audit
   - License compliance checks

3. **Build & Package** (13 targets)
   - Compile for all platforms
   - Create .tar.gz archives
   - Generate .deb packages
   - Create SHA256 checksums
   - Build attestations (v0.25.0+)

4. **Package Manager Publishing**
   - Winget manifest updates (on version tags)
   - Crates.io publication

#### Trigger Events
- Manual dispatch (`workflow_dispatch`)
- Pull requests
- Pushes to master
- Version tags (e.g., `v1.0.0`)

## 4. GitHub Artifact Attestations (2025)

**Blog Announcement:** https://github.blog/news-insights/product-news/introducing-artifact-attestations-now-in-public-beta/
**cargo-dist Integration:** v0.21.0+ (August 2024)

### 4.1 How GitHub Attestations Work

Attestations create **tamper-proof, cryptographically signed records** linking artifacts to their build processes.

#### Technical Foundation
- **Sigstore-based:** Uses open-source Sigstore project
- **Workload identity:** Temporary keypairs instead of long-lived secrets
- **Short-lived certs:** Valid for only 10 minutes
- **GitHub root CA:** GitHub acts as root certificate authority

#### Setup (3 Steps)
```yaml
permissions:
  id-token: write  # Allow ID token creation
  attestations: write  # Allow attestation creation

- name: Attest Build Provenance
  uses: actions/attest-build-provenance@v1
  with:
    subject-path: target/release/myapp
```

#### Verification
```bash
# Using GitHub CLI
gh attestation verify <artifact-path> --repo owner/repo

# Or with downloaded attestation
gh attestation verify <artifact-path> --bundle <attestation.json>
```

### 4.2 cargo-dist Attestation Features

**Recent Updates (v0.30.1, October 2025):**
- GitHub attestations permissions scope moved to job level
- Support for customizing which artifacts are attested
- Can specify attestation phase (build, host, announce)
- Uses glob patterns for artifact selection

#### Configuration
```toml
# Cargo.toml or dist config
[package]
# attestation to create
# none, host, or announce
attestation = "host"

# which artifacts to attest (glob patterns)
attestation-globs = ["*.tar.gz", "*.zip"]
```

#### cargo-dist Attestation Format
- SLSA 1.0 provenance
- Build configuration details
- GitHub Actions workflow information
- Builder identity information

```toml
[package.metadata.dist]
# Targets to build

    "x86_64-unknown-linux-gnu",
    "x86_64-apple-darwin",
    "aarch64-apple-darwin",
    "x86_64-pc-windows-msvc",
]

# Archive formats

# Installers

# Attestations
features = ["attestations"]
```

### 5.6 GitHub Actions Workflow Integration

```yaml
name: Release

on:
  push:
    branches:
      - main
  workflow_dispatch:

permissions:
  contents: write
  pull-requests: write

jobs:
  release-plz:
    name: Release-plz
    runs-on: ubuntu-latest
    outputs:
      created: ${{ steps.release.outputs.created }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Run release-plz
        uses: release-plz/action@v0.5
        with:
          command: release-pr
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Create release
        uses: release-plz/action@v0.5
        with:
          command: github-release
          token: ${{ secrets.GITHUB_TOKEN }}
```

## 6. Pre-Release Validation Strategy

### 6.1 Validation Checklist

```yaml
validate:
  - cargo test --all-features
  - cargo test --no-default-features
  - cargo doc --no-deps --all-features
  - cargo fmt --check
  - cargo clippy --all-targets --all-features -- -D warnings
  - cargo audit
  - cargo-semver-checks (via release-plz)
  - MSRV testing (e.g., 1.56.0)
```

### 6.2 Dependency Management

**Tool:** Dependabot + Mergify

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "cargo"
    directory: "/"
    schedule:
      interval: "daily"
    pull-request-branch-name:
      separator: "/"
    reviewers:
      - "owner"
    assignees:
      - "owner"
```

**Mergify Configuration:**
```yaml
pull_request_rules:
  - name: "Merge Dependabot PRs"
    conditions:
      - author: "dependabot[bot]"
      - "#approved-reviews-by >= 1"
      - "check-success=Tests"
    actions:
      merge:
        method: "squash"
```

## 8. Installation Method Recommendations

### For Users Installing from Source
```bash
cargo install myapp
# Equivalent to: cargo install --registry crates.io --git https://github.com/user/myapp myapp
```

### For Package Manager Users
```bash
# Homebrew
brew install user/tap/myapp

# Winget (Windows)
winget install myapp

# Pacman (Arch Linux)
pacman -S myapp

# apt (Debian/Ubuntu)
apt install myapp
```

### For Direct Binary Downloads
```bash
# Automated installer
curl -fsSL https://github.com/user/myapp/releases/download/latest/install.sh | sh

# Or manual
curl -L https://github.com/user/myapp/releases/download/v1.0.0/myapp-v1.0.0-x86_64-unknown-linux-gnu.tar.gz | tar xz
sudo mv myapp /usr/local/bin/
```

## 9. Supply Chain Security Checklist

- [ ] Enable GitHub Artifact Attestations in workflow
- [ ] Require branch protection with status checks
- [ ] Use personal access tokens (PAT) for auth, not GITHUB_TOKEN where possible
- [ ] Run cargo-audit in CI to detect vulnerable dependencies
- [ ] Enable Dependabot for automated updates
- [ ] Sign Windows binaries (SignPath or similar)
- [ ] Notarize macOS binaries (Apple Developer account)
- [ ] Generate SHA256 checksums for all artifacts
- [ ] Create SBOM (Software Bill of Materials) if needed
- [ ] Document release process in RELEASE-CHECKLIST.md
- [ ] Use semver-checks to prevent accidental breaking changes

## 11. Workflow Summary

### Ideal Release Process (with release-plz + cargo-dist)

1. **Continuous:** Developers use Conventional Commits in PR messages
2. **Automated:** release-plz creates Release PR weekly (configurable)
3. **Review:** Maintainer reviews version bumps and changelog
4. **Merge:** Merge Release PR to main
5. **Automated:**
   - release-plz creates git tag
   - cargo-dist builds 13 platform binaries
   - Creates GitHub release with artifacts
   - Publishes to crates.io
   - Updates Homebrew tap
   - Generates attestations
   - Updates documentation
6. **Done:** Zero manual steps post-merge

### Time Investment
- Initial setup: 2-4 hours
- Per-release: 5 minutes (review and merge)
- Manual build/distribution: Eliminated
